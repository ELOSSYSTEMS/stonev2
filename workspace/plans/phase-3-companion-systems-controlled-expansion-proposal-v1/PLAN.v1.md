## Status

- `Verified`: This package is documentation-only.

- `Verified`: Its purpose is to determine whether Stone V2 should reopen companion-family planning after the bounded shoppable-media runway was judged locally exhausted.

## Compared Options

### Option A, reopen Phase 3 companion planning through one collection-companion boundary first

- `Inference`: Reopen companion-family planning, but keep it sequenced family-by-family rather than pretending both companion families can be reopened together safely.
- `Inference`: Start with `collection companion` because it carries lower commerce pressure than `PDP companion`, aligns with the family ledger's lighter collection-support role, and can define template-coupled limits before product-detail risk enters.
- `Verified`: Keep the immediate next unit docs-only.

### Option B, reopen Phase 3 companion planning through one PDP-companion boundary first

- `Inference`: Start with `PDP companion` because it is the stronger product-detail family.
- `Inference`: This is not the most honest first move because PDP reopening sits closer to buy-box, pricing, recommendation, gallery, and heavier commerce adjacency than collection reopening.
- `Inference`: Starting here would make it easier to smuggle richer commerce assumptions into a supposedly bounded planning step.

### Option C, open one shared broader companion-systems boundary for collection and PDP together

- `Inference`: Treat companion systems as one combined planning problem.
- `Inference`: This is too coarse because the ledger keeps `collection companion` and `PDP companion` as distinct families, and the template-coupled risks are materially different.
- `Inference`: A shared boundary would hide sequencing, surface-budget, and commerce-distinctness questions that need to stay explicit.

### Option D, do not reopen companion-family planning and defer directly to preset-pack sequencing

- `Inference`: This is the safest fallback.
- `Inference`: It is not the strongest evidence-based choice because Phase 3 still explicitly includes collection and PDP companion families, and preset planning should consume resolved family-boundary decisions rather than substitute for them.

## Recommendation

- `Verified`: **Recommend Option A.**
- `Inference`: Stone V2 should reopen companion-family planning as a broader Phase 3 track only in a fail-closed, family-by-family form, with `collection companion` going first and `PDP companion` remaining blocked until a later decision.

4. `Verified`: `UNIT-074` concluded that deeper collection-aware runtime work should not resume automatically, but it also identified collection as the lighter companion family before the repo moved into PDP-aware work.
5. `Verified`: `UNIT-075` through `UNIT-078` proved that PDP companion work is viable only in a very narrow bounded form and then explicitly paused further runtime expansion because richer product-detail behavior would need a broader planning class.
6. `Inference`: A collection-first broader boundary is therefore the safer re-entry point because it preserves the lower-risk side of companion planning while keeping PDP-adjacent commerce pressure fail-closed.
